# ADR 003: Выделение слияния YAML в отдельный субмодуль

## Дата
2026-04-01

## Статус
Предложено

## Автор
@cherninkiy

## Участники
Команда Protocollab
## Контекст

Модуль `yaml_serializer` в настоящее время содержит пустой файл-заглушку `merge.py` с TODO-комментарием о необходимости реализации трёхстороннего слияния. Основная функциональность модуля — безопасная загрузка, сохранение и отслеживание изменений YAML-файлов с поддержкой `!include` и сохранением комментариев.  

Трёхстороннее слияние — сложная задача, включающая:
- Рекурсивное слияние YAML-узлов (`CommentedMap`, `CommentedSeq`, скаляров);
- Сохранение комментариев и форматирования;
- Обработку ссылок `!include` (слияние вложенных деревьев файлов);
- Обнаружение переименований файлов и обновление путей в `!include`;
- Предоставление интерфейса для разрешения конфликтов.

Проект находится на ранней стадии развития, и breaking changes допустимы. Необходимо принять решение: реализовать слияние **внутри** `yaml_serializer` или **выделить в отдельный субмодуль**.

## Решение

**Выделить функциональность слияния в новый субмодуль `yaml_merger`**, который будет иметь **линейную зависимость от `yaml_serializer`**.

- `yaml_serializer` остаётся самодостаточной библиотекой для безопасной загрузки/сохранения YAML, не имеющей знаний о слиянии.
- `yaml_merger` импортирует `yaml_serializer` и использует его публичное API (типы узлов, утилиты, атрибуты файлов) для реализации трёхстороннего слияния.
- Оба модуля будут располагаться в одном монорепозитории в каталоге `packages/`.
- Пустой файл `merge.py` внутри `yaml_serializer` будет **удалён незамедлительно** (как шаг 0), так как он не используется и заглушка больше не нужна.

### Обоснование

- **Разделение ответственности**: `yaml_serializer` сосредоточен на своей основной задаче — безопасной загрузке, сохранении и отслеживании изменений. Слияние — опциональная, продвинутая функция, которая может развиваться независимо.
- **Гибкость**: Отдельный субмодуль позволяет при необходимости использовать логику слияния с другими YAML-бэкендами, не подключая весь загрузчик.
- **Стабильность**: Пользователи, которым не нужно слияние, не получают лишний код и зависимости.
- **Чистая архитектура**: Направление зависимости естественно (`yaml_merger` → `yaml_serializer`), без циклических связей.
- **Преимущество ранней стадии**: Поскольку код слияния ещё не написан, выделение сейчас позволит избежать breaking changes в будущем.

## Последствия

### Положительные
- `yaml_serializer` становится проще и удобнее в поддержке.
- Логика слияния может разрабатываться, тестироваться и выпускаться независимо.
- Монорепозиторий поощряет модульность и чёткое разделение ответственности.

### Отрицательные
- Два пакета вместо одного: пользователям, которым нужно слияние, придётся устанавливать дополнительный пакет.
- Небольшие накладные расходы на поддержку двух наборов тестов, документации и версионирования.
- Некоторые внутренние функции `yaml_serializer` придётся сделать публичными для использования в `yaml_merger` (например, `canonical_repr`, `compute_hash`, `mark_includes`, `update_file_attr`). Их нужно задокументировать как «внутренние, но стабильные для расширенного использования».

### Смягчение рисков
- Оба пакета остаются в одном монорепозитории с общими инструментами (Poetry workspace), что минимизирует накладные расходы на координацию.
- Можно добавить метод `session.merge()` в `yaml_serializer`, который будет импортировать `yaml_merger` при наличии, обеспечивая единый интерфейс без обязательной зависимости.
- Добавить этап стабилизации `yaml_serializer` перед разработкой `yaml_merger` (см. план реализации).

## План реализации

### Этап 0: Стабилизация `yaml_serializer` (перед выделением)

**Цель:** обеспечить чистый, стабильный публичный API, на который сможет положиться `yaml_merger`.

1. **Удалить `merge.py`** — убрать пустой файл-заглушку.
2. **Ревизия и определение публичного API**:
   - Определить функции, необходимые `yaml_merger` (`canonical_repr`, `compute_hash`, `resolve_include_path`, `is_path_within_root`, `mark_node`, `mark_dirty`, `clear_dirty`, `update_file_attr`, `replace_included`, `mark_includes` и др.).
   - Добавить эти функции в `__all__` в соответствующих модулях.
   - Пометить внутренние функции символом подчёркивания (`_`) для ясности.
3. **Добавить маркеры стабильности** (опционально, через docstring или декораторы):
   - `@stable_api` для функций с гарантией обратной совместимости.
   - `@internal_use_only` для функций, которые могут меняться без предупреждения.
4. **Обновить документацию**:
   - Добавить в `yaml_serializer/README.md` раздел «Стабильность публичного API» со списком стабильных функций.
   - Дополнить docstring описанием поведения и предварительных условий.
5. **Расширить тестовое покрытие** для экспортируемых стабильных функций:
   - Граничные случаи (пустые входные данные, некорректные типы).
   - Обработка ошибок.
   - Корректность возвращаемых значений.
6. **Проверить целостность данных**:
   - Убедиться, что критические атрибуты (`_yaml_file`, `_yaml_parent`, `_yaml_include_path`) корректно сохраняются после операций.
7. **Рефакторинг и чистка**:
   - Удалить неиспользуемый код.
   - Унифицировать стиль (PEP 8, аннотации типов).
   - Гарантировать, что все исключения перехватываются и задокументированы.
8. **Выпустить `yaml_serializer v1.0.0`** с чётко документированным стабильным API и семантическим версионированием.

### Этап 1: Подготовка инфраструктуры

1. Создать `packages/yaml_merger/` с `pyproject.toml`, указав зависимость `yaml_serializer >= 1.0.0`.
2. Настроить корневой `pyproject.toml` как Poetry workspace.
3. Обновить CI для запуска тестов обоих пакетов.

### Этап 2: Реализация `yaml_merger`

1. **Ядро слияния** (`merger.py`):
   - Рекурсивное слияние скаляров, последовательностей и отображений.
   - Стратегия сохранения комментариев (начать с простого выбора левой/правой стороны).
2. **Обработка `!include`** (`include_handler.py`):
   - Рекурсивное слияние включённых файлов с использованием того же алгоритма.
   - Обновление атрибутов `_yaml_file`, `_yaml_include_path`, `_yaml_parent`.
3. **Разрешение конфликтов** (`conflict.py`):
   - Классы данных для описания конфликтов.
   - Встроенные резолверы (`take_left`, `take_right`, `keep_base`, `manual`).
   - Возможность передать пользовательский callback.
4. **Тестирование**:
   - Модульные тесты для каждого компонента.
   - Интеграционные тесты с реальными YAML-файлами и деревьями `!include`.

### Этап 3: Интеграция и документация

1. Добавить метод `SerializerSession.merge()` как адаптер, который при наличии `yaml_merger` загружает три версии и возвращает результат слияния.
2. Обновить документацию `yaml_serializer`, указав на опциональный пакет `yaml_merger`.
3. Написать подробную документацию для `yaml_merger` с примерами и руководством по разрешению конфликтов.

### Этап 4: Релиз

1. Выпустить `yaml_merger v1.0.0`, совместимый с `yaml_serializer v1.0.0`.
2. Опционально выпустить мета-пакет (например, `protocollab-yaml`), включающий оба модуля.

## Рассмотренные альтернативы

### Реализовать слияние внутри `yaml_serializer`
- **Плюсы**: Один пакет, нет дополнительных зависимостей.
- **Минусы**: Раздувание модуля, более тесная связанность, усложнение независимого развития, будущее выделение будет breaking change.

### Создать полностью независимую библиотеку слияния со своей абстракцией YAML
- **Плюсы**: Возможность поддержки разных YAML-бэкендов.
- **Минусы**: Избыточно для текущих потребностей, приведёт к дублированию логики, уже реализованной в `yaml_serializer` (работа с комментариями, `!include`). Не оправдано.
