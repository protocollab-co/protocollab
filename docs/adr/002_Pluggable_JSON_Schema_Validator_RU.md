# ADR 002: Создание подключаемого модуля валидации JSON Schema

## Статус
Предложено

## Контекст
В рамках проекта `protocollab` используется валидация YAML-спецификаций по JSON Schema (базовая и строгая). Сейчас валидация реализована через прямые вызовы библиотеки `jsonschema` в модуле `protocollab.validator`.

Сделаны следующие наблюдения:
- Существуют альтернативные валидаторы (`fastjsonschema`, `jsonscreamer`), которые могут быть значительно быстрее или безопаснее.
- Разные сценарии использования предъявляют разные требования: в CI/CD важна скорость, при обработке непроверенных схем — безопасность, при сложных схемах — максимальная совместимость со стандартом.
- Желательно, чтобы модуль валидации можно было использовать независимо от `protocollab` (например, в других проектах).
- Решение должно оставаться расширяемым для будущих сценариев использования, включая возможную поддержку разных версий JSON Schema и других форматов описания, таких как OpenAPI и AsyncAPI.

## Решение
Мы создадим **отдельный, переиспользуемый Python-модуль** `jsonschema_validator` (или с похожим именем), который будет предоставлять единый интерфейс для валидации JSON Schema с подключаемыми бэкендами. Модуль будет:

1. **Определять абстрактный интерфейс валидации** и унифицированную модель результата валидации для независимого от бэкенда вывода. Точные имена классов намеренно не фиксируются на стадии ADR, поскольку итоговое именование должно быть согласовано с уже существующими типами `protocollab.validator` и может измениться в зависимости от окончательного набора бэкендов и формы интеграции.
2. **Реализовывать адаптеры** как минимум для трёх бэкендов JSON Schema:
   - `jsonschema` – максимальная совместимость, базовый бэкенд.
   - `fastjsonschema` – высокая производительность (опционально, использует `exec`).
   - `jsonscreamer` – высокая производительность без `exec` (опционально).
3. **Рассматривать в будущем интеграцию с Pydantic** для валидации на основе моделей как отдельное направление расширения, а не как часть списка JSON Schema бэкендов.
4. **Предоставлять фабрику** (`ValidatorFactory.create(backend="auto", cache=True, **options)`), которая выбирает доступный бэкенд с предложенным порядком по умолчанию `jsonscreamer` -> `jsonschema`. Поскольку `fastjsonschema` использует `exec`, режим `auto` не должен выбирать его по умолчанию для сценариев с недоверенными схемами; его использование должно требовать явного выбора бэкенда или другого явного opt-in механизма, который будет определён на этапе реализации.
5. **Включать кэширование** скомпилированных валидаторов (по хешу схемы и версии бэкенда), чтобы избежать повторных накладных расходов на компиляцию.
6. **Нормализовать сообщения об ошибках** разных бэкендов к общему внутреннему формату. Внешний пользовательский формат путей ошибок должен оставаться совместимым с текущим стилем dot-notation, который уже используется в `protocollab` (например, `meta.id` или `seq[0].type`), даже если внутренний слой нормализации использует другое представление.
7. **Интегрироваться в `protocollab`**: заменить прямые вызовы `jsonschema` новым модулем и добавить CLI-опцию `--validator-backend` для команды `validate`.
8. **Поставляться как часть монорепозитория `protocollab`** со своими тестами, документацией и опциональными зависимостями, но оставаться полностью независимым (без импортов из `protocollab.*`).

Возможный вариант **будущей** схемы установки через extras (TBD, не реализовано в текущем `pyproject.toml`), например:
```bash
pip install protocollab[validator-jsonschema]   # планируемый минимальный extra с базовым бэкендом
pip install protocollab[validator-all]          # планируемый extra с полным набором бэкендов
```

## Последствия
### Положительные
- Гибкость: пользователи могут выбирать бэкенд под свои задачи.
- Безопасность: возможность избежать `exec` для непроверенных схем.
- Производительность: в CI/CD можно использовать `fastjsonschema` для быстрой валидации.
- Модуль может быть переиспользован в любом Python-проекте.
- Упрощается тестирование (подмена бэкенда, изоляция).
- Снижение вендор-локина для валидации JSON Schema.

### Отрицательные
- Увеличение сложности кода (адаптеры, фабрика, кэш).
- Необходимость поддерживать несколько бэкендов и следить за их совместимостью.
- Увеличение размера дистрибутива (хотя бэкенды опциональны).
- Возможные расхождения в поведении разных бэкендов, что потребует документирования и, возможно, обработки исключений.

## Альтернативы
1. **Оставить как есть (только `jsonschema`)**  
   – Не требует дополнительных усилий, но лишает гибкости и возможностей оптимизации.

2. **Выделить модуль, но без адаптеров (только `jsonschema`)**  
   – Даёт независимость, но не решает проблему выбора бэкенда.

3. **Использовать `fastjsonschema` по умолчанию, а `jsonschema` как fallback**  
   – Проще, но не даёт пользователю контроля и может быть небезопасно при непроверенных схемах.

4. **Полностью заменить на `jsonscreamer`**  
   – Улучшит скорость и безопасность, но может потерять совместимость со сложными схемами.

**Почему выбрано текущее решение:**  
Оно даёт максимальную гибкость, позволяет использовать сильные стороны разных библиотек и соответствует философии модульности проекта. Реализация адаптеров не слишком сложна, а выгоды перевешивают издержки.
