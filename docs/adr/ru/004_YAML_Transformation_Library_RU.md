# ADR 004: Библиотека трансформации YAML и архитектура ProtoCollab

## Статус

Предложено (заменяет [ADR 003](003_Separate_YAML_Merge_Submodule_RU.md))

## Дата

2026-04-17

## Автор

@cherninkiy

## Принимающие решение

Сопровождающие Protocollab

## Контекст

Пакет `yaml_serializer` обеспечивает безопасную загрузку, сохранение и отслеживание изменений YAML-файлов с поддержкой `!include` и сохранением комментариев. В настоящее время он не содержит функциональности слияния (заглушка `merge.py` удалена).

Мультиагентная система (спецификатор, оптимизатор, тестировщик, слиятель, рецензент, генератор), совместно работающая над репозиторием YAML-спецификаций, требует **широкого набора YAML-трансформаций**, выходящего за рамки простого слияния:

- Трёхстороннее слияние с разрешением конфликтов (включая помощь LLM)
- Фильтрация подмножеств спецификации (например, для тестирования)
- Подстановка переменных (`${VAR}`) в строках и ключах
- Переименование ключей с сохранением комментариев
- Нормализация (сортировка ключей, канонизация типов, удаление дубликатов)
- Применение JSON Patch (RFC 6902)
- Установка/удаление значений по JSONPath
- Сравнение двух YAML-деревьев (diff)
- Извлечение поддеревьев, разделение одного документа на несколько
- Поверхностное и глубокое копирование YAML-узлов с сохранением тегов `!include` и комментариев

Все эти операции должны сохранять комментарии, форматирование и метаданные `!include`, чтобы оставаться совместимыми с циклом загрузки-сохранения `yaml_serializer`. Они также должны интегрироваться с отслеживанием изменений `yaml_serializer` (`mark_dirty`, `_yaml_parent`).

Проект находится на ранней стадии; допустимы критические изменения. Нам необходимо определить архитектуру возможностей трансформации YAML и их связь с вышележащим слоем `ProtoCollab`.

## Решение

Мы создадим отдельный пакет **`yaml_transformer`**, предоставляющий полный набор утилит трансформации YAML, включая трёхстороннее слияние. Этот пакет будет иметь **линейную зависимость от `yaml_serializer`** и будет использоваться слоем `ProtoCollab`.

Дополнительно мы определяем `ProtoCollab` как **высокоуровневый архитектурный слой**, предоставляющий унифицированный API для агентов и пользователей для взаимодействия с репозиторием YAML-спецификаций. `ProtoCollab` будет использовать:

- `yaml_transformer` для всех операций, специфичных для YAML.
- **Подключаемый бэкенд репозитория** (интерфейс `RepositoryBackend`) для контроля версий и доступа к файлам (подробно описано в ADR 005).

Таким образом, цепочка зависимостей выглядит так:

```
ProtoCollab → yaml_transformer → yaml_serializer
```

### Что включает `yaml_transformer`

#### Мутирующие трансформации (работают с `CommentedMap`/`CommentedSeq`, вызывают `mark_dirty`)

- `merge(docs, strategy)` – трёхстороннее слияние с отчётом о конфликтах.
- `filter(data, condition)` – оставить только соответствующие элементы (точное совпадение dict или JMESPath).
- `substitute(data, variables, syntax)` – замена переменных в строках и ключах.
- `rename(data, mapping)` – переименование ключей словаря с сохранением комментариев.
- `normalize(data, sort_keys, canonical_types, remove_duplicates)`.
- `patch(data, patches)` – применение операций JSON Patch (RFC 6902).
- `set_value(data, path, value, create_parents)` – установка по JSONPath.
- `delete(data, path)` – удаление по JSONPath.

#### Немутирующие утилиты

- `diff(left, right)` – возвращает различия (на основе `deepdiff`).
- `extract(data, path)` – получить поддерево (возвращает живую ссылку; для отсоединения используйте `copy_node`).
- `split(data, by, path)` – разделить один документ на несколько. Комментарии, прикреплённые к разделяемым узлам, можно оставить в исходном документе, переместить в новый или скопировать в оба. Стратегия `both` дублирует комментарии; последующее трёхстороннее слияние способно согласовать эти дубликаты.
- `flatten(data)` – создаёт глубокую копию без метаданных `!include` (теги разрешены и удалены).
- `copy_node(node, mode: CopyMode = CopyMode.REFERENCE, new_root_dir: Path | None = None) -> CommentedMap | CommentedSeq`  
  Низкоуровневый примитив, создающий копию YAML-узла в соответствии с выбранным `CopyMode`. Используется внутри `TransformBuilder.copy()` и доступен для прямого использования.

#### Ленивый `TransformBuilder`

- Накапливает операции без их немедленного выполнения.
- Применяет их с помощью:
  - `mutate()` – на месте, помечает грязным через распространение `mark_dirty`. **Не атомарно**: если операция в середине цепочки завершится ошибкой, документ останется частично изменённым. Для атомарности сначала применяйте изменения к копии.
  - `copy(mode)` – создаёт новый документ в соответствии с выбранным `CopyMode` после применения всех накопленных операций.

### Семантика копирования с учётом `!include`

Поскольку `yaml_serializer` ограничивает разрешение `!include` корневой директорией, копирование YAML-документа требует аккуратного обращения с метаданными файловой системы (`_yaml_file`, `_yaml_include_path`). Мы определяем три режима копирования, доступные через `TransformBuilder.copy(mode)` и низкоуровневую функцию `copy_node`.

#### 1. `REFERENCE` (поверхностное копирование)

- **Поведение**: Создаёт новый корневой узел, но сохраняет ссылки на все дочерние узлы (включая полученные из `!include`). Метаданные не изменяются.
- **Сценарий использования**: Временные структуры в памяти в пределах той же корневой директории.
- **Ограничение**: Сохранение копии в другую корневую директорию нарушит пути `!include`. Вместо этого используйте `DETACHED` или `FLATTENED`.
- **Отслеживание изменений**: Модификации через копию помечают исходный корень как грязный.

#### 2. `DETACHED` (глубокое копирование с сохранением тегов `!include`)

- **Поведение**: Рекурсивно создаёт новые узлы, но теги `!include` сохраняются (не разрешаются). Метаданные обновляются в соответствии с новой корневой директорией (`new_root_dir`). Перед копированием проверяется, что все пути `!include` находятся внутри `new_root_dir`.
- **Обновление метаданных**:
  - `_yaml_file` устанавливается в новый путь к файлу внутри `new_root_dir`.
  - `_yaml_include_path` пересчитывается как относительный путь от нового родителя.
- **Опциональное копирование файлов**: Если `copy_files=True`, все включаемые файлы физически копируются в `new_root_dir` с сохранением относительной структуры. Конфликты имён разрешаются через `yaml_transformer.merge`.
- **Сценарий использования**: Миграция спецификации в другой репозиторий с сохранением модульности.

#### 3. `FLATTENED` (полное разрешение, независимость от файловой системы)

- **Поведение**: Рекурсивно заменяет каждый узел `!include` его содержимым. Результат — самодостаточный YAML-документ без тегов `!include` и метаданных файловой системы.
- **Контроль рекурсии**: `max_include_depth` и обнаружение циклов через стек загрузки.
- **Сохранение комментариев**: Комментарии из включаемых файлов сохраняются в плоском дереве.
- **Сценарий использования**: Передача полной спецификации агенту, не имеющему доступа к файловой системе; создание распространяемого снимка.

#### Гарантии валидности

| Режим       | Условие                                                                 | Действие при нарушении                      |
|-------------|-------------------------------------------------------------------------|---------------------------------------------|
| `REFERENCE` | Копия не сохраняется в другую корневую директорию.                      | `ValueError`                                |
| `DETACHED`  | Все пути включения разрешимы относительно `new_root_dir`.                | `FileNotFoundError` с указанием проблемного пути |
| `FLATTENED` | Все включаемые файлы доступны для чтения в момент копирования.           | `IncludeError`                              |

### Трёхстороннее слияние (TWM)

Трёхстороннее слияние — это основной механизм разрешения конфликтов, используемый как непосредственно агентами, так и косвенно для конфликтов имён файлов при копировании `DETACHED`.

```python
def merge(
    base: CommentedMap | CommentedSeq,
    left: CommentedMap | CommentedSeq,
    right: CommentedMap | CommentedSeq,
    strategy: MergeStrategy = MergeStrategy.DETERMINISTIC,
) -> MergeResult:
    ...
```

- **Обнаружение изменений**: Использует хеш-функцию, **включающую комментарии** (расширенная `compute_hash_with_comments`, построенная на примитивах `yaml_serializer`). Это гарантирует, что изменения комментариев рассматриваются как изменения и могут вызывать конфликты.
- **Обработка `!include`**: Узлы с `!include` сравниваются по метаданным (путь и хеш файла). Содержимое включаемого файла не сливается неявно; при необходимости слияния содержимого пользователи должны сначала применить `flatten` к документам.
- **Стратегии разрешения конфликтов** (MVP):
  - `DETERMINISTIC`: Разрешить с использованием детерминированного правила, но пометить результат как конфликтный.
  - `FAIL_FAST`: Выбросить `MergeConflictError` при первом конфликте.
  - *Примечание: `MARK_CONFLICTS` (маркеры в стиле Git) исключена из начальной версии, так как порождает невалидный YAML и усложняет цикл загрузки-сохранения.*
- **Результат**: `MergeResult` содержит слитый документ, список объектов `Conflict` и флаг `had_conflicts`.

TWM также служит основой для согласования дублированных комментариев, созданных `split` с режимом `both`, и для разрешения коллизий имён файлов при `copy_files=True` в режиме `DETACHED`.

### Почему не отдельный `yaml_merger`?

- Выделенный `yaml_merger` был бы слишком узким; агентам также нужны фильтрация, подстановка, патчинг и т.д.
- Разделение на `yaml_merger` + `yaml_transformer` вынудило бы агентов зависеть от двух пакетов и дублировать низкоуровневые утилиты (например, `copy_node`).
- Единый `yaml_transformer` более связный и лёгкий в сопровождении.

### Почему не оставить всё в `yaml_serializer`?

- `yaml_serializer` остаётся сосредоточенным на безопасной загрузке, сохранении и отслеживании изменений.
- Логика трансформаций сложна и может развиваться независимо.
- Пользователи, которым нужна только загрузка/сохранение, не вынуждены тянуть код трансформаций.

## Последствия

### Положительные

- Чёткое разделение ответственности: `yaml_serializer` (базовый ввод-вывод), `yaml_transformer` (трансформации), `ProtoCollab` (оркестрация + бэкенды).
- Мультиагентная система получает единую мощную библиотеку для всех манипуляций с YAML.
- Ленивое вычисление (`TransformBuilder`) улучшает производительность и позволяет выбирать мутацию/копирование в конце.
- Подключаемые бэкенды репозитория (ADR 005) делают `ProtoCollab` адаптируемым к Git, многоагентному хранилищу или простым файлам.

### Отрицательные

- Три пакета вместо одного: пользователи, нуждающиеся в трансформациях, должны установить `yaml_transformer`; для полной коллаборации нужен также `ProtoCollab`.
- Небольшие накладные расходы на сопровождение трёх наборов тестов и документации.
- `yaml_serializer` должен предоставить дополнительные стабильные API-функции: `copy_node`, `propagate_dirty_up`, а также существующие утилиты (`canonical_repr`, `compute_hash`, `mark_dirty` и т.д.).

### Смягчение

- Держать все пакеты в одном монорепозитории с общим инструментарием (Poetry workspace).
- Предоставить мета-пакет `protocollab`, объединяющий `yaml_serializer`, `yaml_transformer` и `ProtoCollab`.
- `yaml_serializer` уже стабилен (v1.0.1), и требуемые API либо присутствуют, либо могут быть добавлены обратно совместимым образом.

## План реализации

### Фаза 1: Создание пакета `yaml_transformer`
1. Создать `src/yaml_transformer/` с `pyproject.toml`, зависящим от `yaml_serializer >= 1.0.1`.
2. Реализовать основные трансформации (`transform.py`, `merge.py`, `diff.py`, `builder.py`).
3. Реализовать `TransformBuilder` с ленивым выполнением и режимами копирования.
4. Написать обширные тесты (модульные + интеграционные с реальными YAML-файлами и `!include`).
5. Выпустить `yaml_transformer` v1.0.0.

### Фаза 2: Реализация `ProtoCollab` (ADR 005)
1. Определить абстрактный класс `RepositoryBackend` в новом подпакете `protocollab.yaml_repository`.
2. Реализовать `ManualBackend`, `MultiAgentBackend` и Git-бэкенды.
3. Построить высокоуровневый API `ProtoCollab`, используя `yaml_transformer`.
4. Предоставить фабрику `create_repository_backend()` в пакете `protocollab`.

## Рассмотренные альтернативы

- **Оставить слияние внутри `yaml_serializer`** – отвергнуто, так как это раздует ядро и не покроет другие необходимые трансформации.
- **Создать отдельные `yaml_merger` и `yaml_transformer`** – отвергнуто из-за дублирования и фрагментации.
- **Использовать `gitpython` напрямую внутри `ProtoCollab` без абстракции бэкенда** – отвергнуто, так как это жёстко привяжет к Git и предотвратит многоагентное хранилище (рассмотрено в ADR 005).
