# ADR 001: Замена глобального состояния явными сессиями и улучшение гибридного дизайна загрузчика

## Статус
Принято

## Контекст
Модуль `yaml_serializer` в настоящее время использует глобальный контекст `_CTX`, который хранит все загруженные файлы, конфигурацию и состояние парсинга. Такой дизайн приводит к:

- **Отсутствию потокобезопасности** – конкурентные операции разрушают контекст.
- **Отсутствию изоляции** – разные части приложения не могут загружать независимые YAML-деревья без взаимовлияния.
- **Неограниченному росту памяти** – все загруженные файлы остаются в памяти навсегда.
- **Сложностям в тестировании** – тесты должны вручную сбрасывать глобальное состояние.
- **Негибкости** – невозможно использовать разные лимиты безопасности для разных загрузок.

Модуль `protocollab.loader` уже реализует гибридный подход: глобальный загрузчик по умолчанию с `MemoryCache` для удобства, а также возможность создания изолированных экземпляров. Однако у него есть свои недостатки:

- Глобальный загрузчик недоступен публично, поэтому очистка кэша или его проверка невозможны.
- `MemoryCache` не имеет ограничения размера – он может неограниченно расти.
- Потокобезопасность не задокументирована, а глобальный загрузчик небезопасен для конкурентного использования.
- Нет простого способа перенастроить глобальный загрузчик (например, увеличить размер кэша или изменить лимиты безопасности) без переопределения.

Необходимо решить эти проблемы, чтобы обе библиотеки были готовы к использованию в долгоживущих сервисах и многопоточных средах.

## Решение

### 1. yaml_serializer: явный паттерн сессии
Заменим глобальный `_CTX` на явный класс `SerializerSession`, который инкапсулирует всё состояние и операции.

- **`SerializerSession`** будет иметь методы `load`, `save`, `rename`, `propagate_dirty` и `clear`. Конструктор принимает конфигурационный словарь.
- Старые функции верхнего уровня (`load_yaml_root`, `save_yaml_root` и др.) **удаляются** как breaking change. Поскольку библиотека ещё не достигла стабильного релиза (v1.0), чистый разрыв предпочтительнее поддержки устаревших обёрток.
- Все внутренние помощники (например, `include_constructor`, `include_representer`) будут переработаны так, чтобы принимать ссылку на сессию (хранящуюся в экземпляре YAML) вместо использования `_CTX`.
- Такой дизайн обеспечивает:
  - **Изоляцию**: разные сессии не влияют друг на друга.
  - **Потокобезопасность**: сессии не разделяются по умолчанию; каждый поток может создать свою.
  - **Управление ресурсами**: сессии можно удалять, освобождая память.
  - **Гибкость**: разные лимиты безопасности для разных сессий.

### 2. protocollab.loader: улучшенный гибридный дизайн
Сохраним гибридную модель, но устраним недостатки:

- **Сделаем глобальный загрузчик доступным**: добавим функцию `get_global_loader()`, возвращающую экземпляр загрузчика по умолчанию.
- **Ограничим размер кэша**: расширим `MemoryCache`, добавив параметр `max_size` (максимальное количество записей) и реализуем LRU-вытеснение с помощью `OrderedDict`. Глобальный загрузчик можно будет настраивать через новую функцию `configure_global(max_cache_size, config)`.
- **Документируем потокобезопасность**: явно укажем, что глобальный загрузчик не является потокобезопасным; для многопоточных нагрузок рекомендуется использовать отдельные экземпляры `ProtocolLoader`.
- **Настраиваемый глобальный загрузчик**: предоставим `configure_global()` для переинициализации глобального загрузчика новыми настройками без нарушения работы существующего кода, ожидающего `_default_loader`.

Эти улучшения сохраняют удобство однострочного вызова для простых скриптов и дают опытным пользователям необходимый контроль.

## Последствия

### Положительные
- **Потокобезопасность**: обе библиотеки можно безопасно использовать в многопоточных приложениях, создавая сессии/загрузчики на поток.
- **Контроль памяти**: кэши могут быть ограничены по размеру, предотвращая неограниченный рост.
- **Изоляция**: несколько независимых задач могут сосуществовать без взаимовлияния.
- **Тестируемость**: unit-тесты могут создавать свежие сессии/загрузчики без загрязнения между тестами.
- **Гибкость**: разные части приложения могут использовать разные настройки безопасности или стратегии кэширования.
- **Обратная совместимость (loader)**: существующий код, использующий стандартный API `protocollab.loader`, продолжает работать, а продвинутые пользователи получают явный доступ к конфигурации общего загрузчика.

### Отрицательные
- **Усилия по миграции (serializer)**: пользователи `yaml_serializer` должны обновить код, вызывавший удалённые функции верхнего уровня, и перейти на `SerializerSession` перед обновлением.
- **Небольшое увеличение объёма кода** в некоторых сценариях (явное создание сессии), но удобство глобального загрузчика сохраняется для простых случаев.
- **Возможная путаница** при выборе между глобальным и явным экземпляром; документация будет чётко объяснять компромиссы.

## Связанные задачи
- Глобальный контекст в `yaml_serializer` → препятствует использованию в серверных окружениях.
- Утечка памяти и отсутствие управления кэшем в `protocollab.loader`.

## План реализации
Оба рефакторинга будут реализованы в отдельной ветке:
- `refactor/remove-global-context`

Документация будет обновлена с учётом новых API и рекомендаций.
